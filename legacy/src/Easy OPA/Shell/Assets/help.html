Introduction

The purpose of this program is to facilitate the conversion of ILR input files from one year to the next for the purposes of testing calculation and validation rules. Ultimately it is the intent of this program to provide a full cradle to grave test environment where files can be converted, saved and introduced to the testing API; the results of which become available for review at some later point when processing is complete.

Simple Application Usage

The simplest example of usage is:
  1. Select a conversion configuration, noting the supported years.
  2. Select an input file matching the input requirements of the selected configuration
  3. Run conversion
  4. Review the operation outcome.

The process is inductive; so buttons won't enable until an outstanding operation is completed.

Select Conversion

The process begins by selecting a ‘conversion configuration’. This is a pre configured list of default conversions applicable to the ILR files being converted; at the time of writing there is only one valid conversion available for converting 16-17 to 17-18 ILR files.

A conversion configuration carries a number of prerequisites and settings with it that will be applied during loading. It has the following prerequisites:
  • the input file must conform to the published schema definition for the source year
  • loading the (16-17 to 17-18) configuration applies 24 default data transformations that...
    • includes the rolling forward of a around 18 dates held within the message
    • and excludes the collection (harvesting) of some incoming values as their content fails to provide the user with any viable pattern of recognition that can be used during custom transformation.
    • applies a custom transformation that will apply a value using a viable range to a new property (Learning Delivery Work Placement Hours) appearing in this years ILR.
  • the output file generated is tested and must conform to the published schema for the target year.

Select Input File

Selecting an input file invokes the 'open file' dialog. You will be directed to load an XML file. The loading process will verify the content of the selected file against the source schema requirements set out in the configuration. If the file is invalid you will be notified; assuming it is OK, the loading process continues with 'value harvesting'. Value harvesting takes place to enable custom data transformations based on 'real value' operations.

Run Conversion

The conversion process will generate the output file using default, edited or loaded data transformations. The output file is currently deposited in the same location as the source file with an amended file name. The output file is checked against the target output configuration schema and you will be notified if there are any issues with it.

Advanced Application Usage

Data Transformations

The act of generating an output file passes all incoming data through a 'transformation process' before it ends up in the output. It is the managment of this intermediate process that offers the user the opportunity to apply custom transformations. The data elements available for transformation are encapsulated in items referred to as 'mapping conventions'. There are currently 26 mapping conventions offering access to over 90 data elements. The operations applied to these data elements are referred to as 'transformation overlays'. The types of operations available in each overlay is based upon it's mapping mode and data type.

Data types

  • Dates
    • These include system dates such as file preparation dates as well as business specific dates (Date of Birth, Start and End dates, etc).
  • Everything else
    • Everyting else covers numbers, codes, pure 'strings' as in unformatted arrays of alpha-numerics.

Mapping Modes

  • Target only
    • Data elements in mapping conventions initialise in this mode. Target only, as it suggests, is where an element has no source from which to draw transformation data.
  • Matched
    • Data elements are matched between source and destination messages.
The loaded configuration designates which of the mapping conventions are relevant for the required conversion. The configuration enables access to mapped conventions allowing the user to customise a transformation.

Data elements on mapping conventions commence life in 'target only' mode. Target only mode caters for a smaller subset of available transformation operations than the fuller 'matched mode' operations.
When the conversion configuration is loaded the source types are compared to the target types and where an entity to element match is achieved the mapping operation is increased to 'matched'.

Transformation Operations

  • Pass through
    • This is a matched mode operation where the source data is passed straight through to the destination element.
  • Range
    • Range is designed as a target only operation and it is intended for numeric use.
    • If you have a field that requires a random number value held inside a specific range you can do it using this transformation.
    • An example of it's use is for "learning delivery work placement" where placement hours should be set to a value inside a range of 1 to 12 for the 17-18 ILR.
    • Two numbers are expected for this operation. If you add more than two, only the first two are used. If any of the first two elements are anything other than pure numbers, the result will be a mapping failure that will look like this:

  • Random
    • Random is also intended as a target only operation where you can offer a set of values from which the value set will be randomly picked for each instance.
  • Replace
    • Replace is a matched mode operation where the comparators are selected from harvested values. you can accumulate you comparators from multiple harvests by repeating the 'select input file' process.
    • The comparators contain items for replacment; operands contain the replacements. If the relationship between comparator and operand is 1:1 or n:1 then the operand is used against every matched comparator during transformation. If the relationship between comparator and operand is 1:n then a random selection is made against the operands for each instance found during transformation.

  • Roll forward
    • Will roll dates forward 1 fixed year as long as the resulting date is less than today.
  • Adjust
    • Will adjust dates forward a given amount of months, as long as the resulting date is less than today.

Saving Transformation Sets

If you've spent a lot of time configurating a set of transformations you can save them for later; reducing any effort in involved in repeating a conversion process. It is feasible you may create transformation sets that are specifically designed to induce particular errors during the processing of an ILR submission. When saving a transformation set you will be asked to give it an appropriate description; this will allow you to identify it later, when you want to reload it.

Loading Transformation Sets

The button marked 'Select Transformations' on the main page of the application give you access to saved transformation sets. Loading a selected transformation set loads previously configured transformations without destroying any harvested values.
